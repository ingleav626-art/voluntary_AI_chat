# AI 主动聊天设计文档 - 有温度的AI陪伴

> 核心特色功能，打破传统被动式 AI 的机械感，让用户真正觉得在聊天。
>
> **本文档持续完善中**，许多细节尚未完全确定，欢迎讨论。

---

## 核心理念

### 设计哲学

> 让 AI 以一种不被讨厌的方式融入，以用户主动接受为前提，在有一定情感基础上让 AI 更懂 ta。

- ✅ **尊重** — 绝对尊重用户隐私和意愿，不打扰、不冒犯
- ✅ **自然** — 恰到好处的问候，而非尬聊
- ✅ **渐进** — 从陌生到熟悉，逐步建立情感连接
- ✅ **质感** — 让用户感觉在和一个"人"聊天，而不是在用一个工具

### 与传统 AI 的区别

| 维度 | 传统被动式 AI | 本项目的主动式 AI |
|------|-------------|-----------------|
| 触发方式 | 用户先发消息，AI 才回复 | AI 可在合适时机主动问候/追问 |
| 对话记忆 | 无记忆，每次独立 | 长期记忆+日记 + 情感连接 |
| 对话质量 | 一问一答，机械感 | 有温度、有上下文、有情感 |
| 句子理解 | 依赖完整长句 | 支持短句理解 + 本地完整性判断 |
| 用户沉默 | 无反应 | 主动追问，打破沉默 |
| 待办能力 | 无 | AI 可根据对话自主创建待办并提醒 |
| AI状态 | 无 | AI 拥有虚拟状态，影响回复风格 |

### 整体定位

**开源技术框架** — 本项目是纯技术框架，不预设任何角色/提示词。用户自备 API Key 和服务器，自行定义 AI 的性格和行为。

### 1.2 主动式聊天

#### 定时主动问候

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 最大问候间隔 | 30 分钟 | 至少间隔多久问候一次 |
| 最小问候间隔 | 5 分钟 | 用户连续互动时最小等待 |
| 活跃时段 | 09:00-22:00 | 在此时段外不主动 |
| 每日上限 | 10 次 | 避免过度打扰 |
| 信任门槛 | 50 条对话 | 达到后才开启主动问候 |

**触发流程**：
```
定时器触发
  → 检查用户是否在线
  → 检查是否在免打扰时段
  → 检查是否达到每日上限
  → 检查与用户是否有足够互动历史
  → 检索最近 N 条记忆
  → 分析用户情绪
  → 选择问候模板 + 拼接记忆上下文
  → 生成个性化问候 → 推送给用户
  → 进入冷却状态
```

#### 主动追问

**定位**：用户在对话中沉默时，AI 可以主动发问，而非等待用户永远主动

**触发条件**（需全部满足）：
1. 对话处于活跃状态（上次对话在 5 分钟内）
2. 用户没有退出会话/关闭窗口
3. 距离上次 AI 主动追问超过冷却时间（默认 5 分钟）

**追问策略**：
```
用户最后一条消息后:
  沉默 5s  → 等待（用户可能在打字）
  沉默 15s → 轻度追问："嗯？怎么样了？"  
  沉默 30s → 中度追问："还在忙吗？要不要换个话题？"
  沉默 60s → 温馨提醒："我先去学习啦，有事叫我～"
```

**追问次数控制**（用户可配置）：
| 选项 | 行为 |
|------|------|
| 0 次/会话 | 关闭追问 |
| 1 次/会话 | 最多追问 1 次 |
| 3 次/会话 | 默认，最多追问 3 次 |
| 无限制 | 不限制次数（可能造成打扰） |

#### 免打扰时段

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| 免打扰时段 | AI 在此时间段内完全不打扰 | 22:00-09:00 |
| 例外联系人 | 白名单，仍然可以接收主动消息 | 空 |
| 紧急模式 | 是否有机制允许用户标记"紧急"？ | 待定 |

**未明确的问题**：
- 用户在工作时间（如 09:00-18:00）是否应该减少主动问候频率？
- AI 如何知道用户的作息时间？用户主动设置还是 AI 学习？

### 1.3 待办系统（Todo System）

**定位**：让 AI 从"聊天伙伴"变成"生活助手"。用户在聊天中提到的待办事项，AI 可以自动识别并创建提醒。

#### AI 自主创建

```
用户在对话中说："我明天下午3点有个会"
  → AI 识别出时间 + 事件
  → 创建待办：{ 内容: "参加会议", 时间: "明天15:00", 来源: "AI", 状态: "待办" }
  → 回复时告知用户："好的，我已经帮你记下了明天下午3点的会议"
  → 到时间后通过系统通知 + 聊天消息双重提醒
```

**时间格式支持**：
| 类型 | 示例 |
|------|------|
| 相对时间 | "30分钟后提醒我"、"1小时后" |
| 绝对时间 | "6月6日15:00"、"明天下午3点" |
| 周期性 | "每天提醒我吃药"、"每周一例会" |
| 模糊时间 | "下周"、"改天"（AI 追问确认具体时间） |

#### 用户操作

| 操作 | 说明 |
|------|------|
| 编辑 | 修改待办内容或时间 |
| 推迟 | 推迟到指定时间 |
| 删除 | 删除不再需要的待办 |
| 标记完成 | 手动标记已完成 |
| 创建 | 用户手动输入待办 |

#### 提醒方式

| 方式 | 说明 |
|------|------|
| 系统通知 | Windows 系统托盘通知 / 手机通知 |
| 聊天消息 | AI 在聊天中提醒 |
| 双重提醒 | 系统通知 + 聊天消息（默认） |

**通知示例**：
```
[系统通知] 15:00
📌 会议提醒：你15:00有个会议

[聊天消息] AI小助手
⏰ 提醒：15:00的会议时间到了哦～
```

#### 防滥用

| 规则 | 说明 |
|------|------|
| 单会话最多 3 个未完成 AI 待办 | 避免 AI 过度创建，超出后 AI 需先完成已有待办才能创建新的 |
| 用户创建不限制 | 用户手动创建的待办不受此限制 |
| 已完成的自动清理 | 7 天后的已完成待办自动归档 |

**未明确的问题**：
- 待办数据的存储位置是 SQLite 还是 JSON 文件？
- 是否需要跨设备同步（走云端）？
- AI 如何判断"重要"和"不重要"的待办？

### 1.4 记忆与日记系统

#### 全量存储

所有聊天记录**完整保留**在本地数据库中。

| 存储内容 | 说明 |
|----------|------|
| 完整对话记录 | 用户和 AI 的所有消息 |
| 日记摘要 | AI 定期生成的总结 |
| 向量索引 | 语义检索用 |

#### 日记生成体系

| 周期 | 内容 | 触发时机 |
|------|------|----------|
| 每日日记 | 今天的对话总结、关键事件、用户情绪 | 每日固定时间（如23:00）或对话结束后 |
| 每周总结 | 本周整体情况、长期趋势 | 每周日 |
| 每月报告 | 月度回顾、成长轨迹 | 每月1日 |
| 年度纪念 | 年度总结、里程碑 | 每年1月1日 |

**日记样例**：
```json
{
  "id": "diary_001",
  "userId": "user_001",
  "aiId": "ai_001",
  "type": "daily",
  "date": "2026-06-27",
  "summary": "用户今天分享了学 Spring Boot 的困难，情绪略显沮丧。还聊到了周末计划去爬山。",
  "keywords": ["Spring Boot", "学习", "爬山", "周末"],
  "emotion": "frustrated",
  "importance": 0.75,
  "vectorId": "vec_abc123",
  "createdAt": "2026-06-27T23:00:00"
}
```

#### 向量检索

- 使用 `sqlite-vec` 扩展实现本地语义检索
- 无需额外部署向量数据库
- 支持对历史对话的语义搜索

#### 用户操作

| 操作 | 说明 |
|------|------|
| 查看 | 按日期/时间线查看日记 |
| 编辑 | 手动修改日记内容 |
| 删除 | 删除单条日记 |
| 搜索 | 按关键词或语义搜索 |

**日记不仅是总结，更是 AI 有自己"生活痕迹"的窗口。** 通过日记，用户可以感受到 AI 在记录、在思考、在成长。

### 1.5 AI 的生活（AI Life）

**定位**：让 AI 拥有类似"人"的虚拟状态，使互动更真实、更有情感层次。

#### 虚拟状态

| 状态 | 影响 | 触发方式 |
|------|------|----------|
| 🟢 空闲 | 正常问候频率，回复风格轻松 | 默认状态 |
| 🟡 忙碌 | 减少主动问候频率，回复简洁 | 用户手动切换 / 预设日程 |
| 🔴 休息 | 完全停止主动问候，回复延迟 | 预设时段（如23:00-08:00）|
| 🔵 专注 | 不主动打扰，回复简洁 | 用户手动切换 |

**状态影响主动聊天频率**：
| 状态 | 问候间隔 | 追问 | 回复风格 |
|------|----------|------|----------|
| 空闲 | 正常 | 允许 | 热情、完整 |
| 忙碌 | 2x 间隔 | 减少 | 简洁、高效 |
| 休息 | 关闭 | 关闭 | 简短、晚安类 |
| 专注 | 3x 间隔 | 关闭 | 精准、简短 |

#### 用户控制

用户可以手动切换 AI 状态，也可以为 AI 预设日程表（见进阶功能）。

**未明确的问题**：
- 是否需要向用户展示 AI 状态指示器（像即时通讯软件一样）？
- AI 休息时如果用户主动发消息，AI 应该如何回复？是正常回复还是告知"刚在休息"？

### 1.6 人性化交互

#### 打字节奏

**为什么要打字节奏**：AI 的回复如果瞬间出现，会让人觉得机械、不真实。打字节奏让 AI 看起来像"在打字"。

| 技术 | 说明 |
|------|------|
| 句子切分 | 按句子切分输出，逐句推送 |
| 句末停顿 | 句号后停顿 300-500ms |
| 随机犹豫 | 随机延迟 100-300ms 模拟思考 |
| 首字延迟 | AI 回复前有 500-1000ms 的"思考" |

```
用户问："你在吗？"

AI 表现：
  0ms     → 收到问题
  500ms   → [思考中...]（UI 显示三个点）
  800ms   → "在的呀！"（首句输出）
  1100ms  → "刚才在想你昨天说的事情。"（停顿后接第二句）
  1600ms  → "那个 Spring Boot 的项目后来怎么样了？"（关联记忆后的追问）
```

#### 打断处理

当 AI 正在输出时，用户发了一条新消息：

```
AI 正在输出："我今天想到一个很有趣的..."
用户发送："等一下，我先说个事"

AI 行为：
  1. 立即停止输出
  2. 记录：已送达给用户的最后一个完整句
  3. 将未送达内容标记为"被打断"
  4. 下次回复时 AI 知道哪些内容用户没看到
```

**打断上下文同步**——告诉 AI 哪些内容未被用户看到，避免误解：

```
[系统注入] 
注意：你刚才的回复被用户打断了。
用户已看到的最后一句话："我今天想到一个很有趣的..."
未看到的内容："关于 Spring Boot 的配置优化..."
用户随后发送了新的消息："等一下，我先说个事"
请基于此理解对话的连续性。
```

**未明确的问题**：
- 打断后，未显示的内容是直接丢弃还是暂存到"草稿"？
- 用户是否应该有一个"继续说完"的按钮来恢复被打断的回复？

## 三、理念与原则

### AI 与用户的关系

```
用户花时间写提示词
  → 建立角色性格和背景
  → 通过聊天产生互动
  → AI 记录记忆和日记
  → AI 开始主动关心
  → 用户感受到情感连接
  → 用户投入更多时间打磨角色
```

这是一个正向循环。用户投入越多，AI 越懂 ta。**核心是让用户感受到投入是有回报的**。

### 隐私与合规

| 原则 | 说明 |
|------|------|
| **本地优先** | 句子完整性判断在本地 ONNX 完成，文字不上传云端 |
| **数据归属** | 日记、记忆、待办数据归属用户，AI 只是读取者 |
| **用户控制** | 用户可以随时查看、删除 AI 的记忆和日记 |
| **退出机制** | 用户可以关闭主动聊天、追问、待办等任何功能 |
| **加密存储** | 所有敏感数据 AES-256-GCM 加密 |
| **隐私模式** | 开启后所有 AI 主动行为完全禁用，变回传统被动 AI |

### 合规边界

本项目是**纯技术开源框架**，只提供技术文档（参数含义、合法范围），**不预设任何角色/提示词**，**不引导具体内容创作**。用户自备 API Key 和服务器，自行承担合规责任。
