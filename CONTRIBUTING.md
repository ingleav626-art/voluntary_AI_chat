# 贡献指南

> 本文档规范两人并行开发的协作流程

---

## 开发者角色分工

| 角色 | 负责模块 | 主要职责 |
|------|---------|---------|
| 开发者A | 客户端（JavaFX） | UI界面、WebSocket客户端、本地模型推理 |
| 开发者B | 服务端（Spring Boot） | API接口、数据库、AI代理、消息队列 |

---

## 分支管理

### 分支命名规范
```
feature/<开发者>/<模块>/<功能描述>
fix/<开发者>/<模块>/<问题描述>
```

### 示例
```
feature/A/client/chat-bubble        # 开发者A：客户端聊天气泡
feature/B/server/user-auth          # 开发者B：服务端用户认证
fix/A/client/websocket-reconnect    # 开发者A：修复WebSocket重连
fix/B/server/message-order          # 开发者B：修复消息顺序
```

### 工作流程
1. 从 `main` 创建功能分支
2. 本地开发并测试
3. 推送到远程并创建 PR
4. 至少一位开发者审查后合并
5. 及时删除已合并的分支

## 并行开发规范

### 1. 接口先行
- 在开发新功能前，先定义好接口契约
- 使用 `docs/API.md` 文档化 API
- 客户端和服务端并行开发时，先对齐接口

### 2. 模块隔离
- 客户端和服务端代码完全分离
- 公共模块（common）的修改需要双方确认
- 避免同时修改同一文件

### 3. 通信协议
- WebSocket 消息格式必须统一
- 定义消息类型枚举和数据结构
- 任何协议变更必须通知对方

### 4. 测试要求
- 每个功能必须有对应的单元测试
- 接口变更必须有集成测试
- 提交前运行完整测试套件

## 代码审查清单

### 审查者检查项
- [ ] 代码是否遵循项目规范（AGENTS.md）
- [ ] 是否有适当的异常处理
- [ ] 是否有必要的测试
- [ ] 是否有安全漏洞
- [ ] 是否影响其他模块
- [ ] 文档是否更新

### 提交者自检
- [ ] 本地测试通过
- [ ] 代码风格符合规范
- [ ] 没有硬编码的配置
- [ ] 敏感信息已处理
- [ ] 相关文档已更新

## 常用命令

```bash
# 同步远程分支
git fetch origin
git rebase origin/main

# 运行测试
mvn test

# 代码检查
mvn checkstyle:check

# 构建项目
mvn clean package

# 启动服务端（开发者B）
mvn spring-boot:run -pl server

# 启动客户端（开发者A）
mvn javafx:run -pl client
```

## 问题沟通

### 日常沟通
- 使用 GitHub Issues 跟踪任务
- 复杂问题先在 Issues 中讨论
- 重要决策记录在文档中

### 代码冲突解决
1. 优先理解冲突双方的意图
2. 复杂冲突必须沟通确认
3. 解决后必须完整测试
4. 记录冲突解决方案
